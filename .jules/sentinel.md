## 2025-02-23 - Fix SSRF IPv6 Bypass
**Vulnerability:** The application's SSRF protection allowed an attacker to bypass private IP validation by supplying an IPv6 literal with brackets (e.g., `http://[fe80::1]/`). The code's `dns.promises.lookup()` would normally crash with `ENOTFOUND` when passed bracketed hostnames, effectively acting as an accidental block for IPv6 literals. However, if this crash was "fixed" by simply stripping brackets without simultaneously patching the IP validation regexes, the regexes would fail to catch IPv6 local-link (e.g., `fe80::/10`) or IPv4-mapped IPv6 addresses, opening a direct SSRF vulnerability.
**Learning:** Fixing one bug (an unhandled network error) can easily create a critical security vulnerability if the underlying defense-in-depth mechanisms (like IP validation regexes) are not comprehensive. The regex `^f[cd][0-9a-f]{2}:` only covered Unique Local Addresses (`fc00::/7`), completely missing Link-Local Addresses (`fe80::/10`).
**Prevention:** When stripping formatting characters (like IPv6 brackets) to "fix" a parser error, always verify that the downstream security checks are actually capable of securely handling the newly unformatted input. In this case, adding `/^fe[89ab][0-9a-f]:/i` was required to block Link-Local Addresses securely.
